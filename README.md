# Quant Project 说明文档

本项目用于支持多市场量化因子研究与回测，当前重点围绕“基础数据沉淀 -> 单项研究 -> 因子打分 -> 回测验证 -> 有效因子沉淀”这条主线展开。

项目整体目标不是一开始做一个大而全的超级数据库，而是建立一个可复用、可扩展、便于逐步积累的研究框架。

## 1. 项目结构

当前建议的目录结构如下：

```text
Quant Project/
├─ backtest/
├─ base_layer/
│  ├─ a_share/
│  ├─ hk_share/
│  └─ us_share/
├─ data_fetch/
│  ├─ a_share/
│  ├─ hk_share/
│  ├─ us_share/
│  └─ futu_common/
├─ factors/
├─ research/
├─ .env
└─ requirements.txt
```

各目录职责如下：

- `base_layer/`
  用于存放各市场的基础层数据，是研究和回测的公共数据底座。

- `data_fetch/`
  用于存放各市场的数据拉取函数。当前已经实现了 A 股的基础拉取函数，后续港股、美股也建议按相同方式扩展。

- `backtest/`
  用于存放各市场对应的回测函数。回测函数应尽量只依赖统一的因子打分表和对应市场的回测底表。

- `research/`
  用于存放每一个单项研究项目。每个研究都建议单独建立文件夹，避免不同研究之间的脚本、结果和中间数据混在一起。

- `factors/`
  用于存放最终验证有效、可以沉淀复用的因子定义、因子说明、因子实现代码或因子结果。

## 2. 推荐研究流程

本项目推荐使用如下标准流程开展单项因子研究：

1. 在 `research/` 下创建一个新的研究文件夹。
2. 先在 `base_layer/` 中查找是否已经有可直接复用的基础数据。
3. 如果基础数据不足，再到 `data_fetch/` 中寻找现有拉取函数进行补充。
4. 如果现有拉取函数仍不满足需求，可以在项目中新增自定义拉取函数。
5. 以对应市场的“回测底表”作为研究基础底表。
6. 再额外拉取本研究所需的补充指标。
7. 按统一主键进行合表，形成该研究项目自己的研究表。
8. 在研究表上计算因子、做标准化或其他处理，得到统一格式的因子打分表。
9. 将因子打分表与回测底表一起交给 `backtest/` 中对应市场的回测函数执行回测。
10. 对回测结果进行评估，将最终有效因子沉淀到 `factors/` 中。

## 3. 单项研究的组织方式

每一个单项研究都建议在 `research/` 下单独开设文件夹，例如：

```text
research/
└─ a_share_momentum_01/
   ├─ notes.md
   ├─ prepare_data.py
   ├─ build_factor.py
   ├─ run_backtest.py
   ├─ outputs/
   └─ temp/
```

这样做的好处是：

- 每个研究项目边界清晰
- 脚本和结果不容易相互污染
- 便于复盘研究过程
- 便于沉淀有效因子

## 4. 基础层数据的使用方式

`base_layer/` 是本项目的数据公共底层，不建议每次研究都从外部接口完全重新拉取。

推荐原则如下：

- 高频复用的数据，长期维护在 `base_layer/` 中
- 单项研究额外需要的增强字段，按研究需求补充拉取
- 同类增强数据一旦做过，也建议沉淀复用

当前 A 股已明确区分为两类日频表：

### 4.1 A 股回测底表

回测底表用于支撑所有研究项目的交易执行与收益计算，是 A 股研究的基础母表。

当前函数：

- `data_fetch.a_share.fetch_ashare_daily_backtest_base`

当前定位：

- 作为 A 股日线研究与回测的公共基础底表
- 主要提供价格、复权、涨跌停、停牌等高复用字段

### 4.2 A 股技术面补充表

技术面补充表用于承接从回测底表中拆分出来的研究增强指标。

当前函数：

- `data_fetch.a_share.fetch_ashare_daily_technical_supplement`

当前定位：

- 用于补充换手率、量比、市值、估值等研究增强字段
- 适合在具体研究中与回测底表进行合表

## 5. 为什么建议以回测底表作为基础底表

本项目推荐使用“回测底表作为基础层母表”的方式，而不是直接把研究表当成基础层，原因如下：

- 回测底表的复用程度最高
- 几乎所有研究最终都需要落到交易执行与收益计算
- 研究和回测可以共享同一套底层行情口径
- 能避免研究阶段和回测阶段的数据口径不一致
- 可以把非通用的研究增强字段从公共底层中拆出来，保持基础层稳定

也就是说：

- 回测底表是公共基础层
- 研究表是从回测底表出发，再补充项目所需增强字段后形成的项目级数据表

## 6. 研究表的形成方式

每个研究项目都建议按照以下方式构建研究表：

1. 读取对应市场的回测底表
2. 拉取该研究需要的补充指标表
3. 按统一主键合表
4. 得到该研究项目自己的研究表

对于 A 股日频研究，目前建议统一使用以下主键：

- `ts_code`
- `trade_date`

只要是天然日频的研究型表，都建议尽量输出为这组主键，以便后续简单合表。

## 7. 统一版因子打分表

为了让回测函数在不同研究项目之间能够复用，因子打分表建议采用统一格式。

当前推荐的统一版字段为：

- `ts_code`
- `trade_date`
- `signal_score`
- `signal_rank`
- `signal_pct_rank`
- `strategy_id`

各字段含义如下：

- `ts_code`
  股票代码。

- `trade_date`
  该因子信号对应的交易日。

- `signal_score`
  最终信号分数。回测函数最核心依赖的字段，分数越高表示越看好。

- `signal_rank`
  当日横截面排序名次。通常 `1` 表示当日最优。

- `signal_pct_rank`
  当日横截面分位排名，通常在 `0~1` 之间，便于做分层和分组回测。

- `strategy_id`
  因子或策略标识，用于区分不同研究项目或不同参数版本。

### 7.1 统一版因子打分表的设计原则

因子打分表建议尽量只记录“最终信号”，不要重复存储交易约束与底层行情字段。

原因是：

- 交易约束已经存在于回测底表中
- 因子打分表只负责表达“某天某股的最终评价结果”
- 回测函数可以根据因子打分表排序选股，再去回测底表中读取交易执行所需信息

也就是说，因子打分表负责“研究输出”，回测底表负责“交易执行基础数据”。

## 8. 回测函数的输入方式

回测时，建议将以下两类数据共同交给对应市场的回测函数：

1. 对应市场的回测底表
2. 统一格式的因子打分表

回测函数的职责应包括：

- 根据信号分数进行排序、选股或分组
- 按回测规则生成持仓
- 读取回测底表中的价格和交易限制
- 模拟调仓与成交
- 计算净值曲线、收益率、回撤、换手等指标

建议回测函数尽量不要依赖研究增强字段本身，而是只依赖：

- 因子打分表中的统一信号字段
- 回测底表中的交易执行字段

这样回测函数才能在不同研究项目中最大程度复用。

## 9. 关于自定义数据拉取

除现有拉取函数外，本项目也允许研究者在项目中自行编写特殊数据拉取函数。

推荐原则如下：

- 优先复用 `base_layer/` 中已有基础数据
- 优先复用 `data_fetch/` 中已有拉取函数
- 只有在确实需要额外数据时，再编写新的自定义拉取函数
- 新函数尽量与现有命名、主键、保存方式保持一致

如果是日频研究增强型数据，建议尽量输出为：

- `ts_code`
- `trade_date`

这样可以最大程度简化合表。

## 10. 有效因子的沉淀方式

当一个因子经过研究和回测验证后，建议将其沉淀到 `factors/` 中。

沉淀内容建议至少包括：

- 因子名称
- 因子定义
- 因子计算逻辑
- 使用的数据来源
- 回测表现摘要
- 适用市场与适用周期
- 是否已有实现代码

这样后续就可以把 `factors/` 作为项目的因子资产库持续积累。

## 11. 当前推荐的方法论总结

本项目当前推荐的方法论可以概括为：

- 公共高复用数据沉淀在 `base_layer/`
- 每个研究项目在 `research/` 下独立组织
- 研究时以对应市场的回测底表作为基础底表
- 再按研究需要拉取补充指标并合表
- 因子研究完成后，产出统一格式的因子打分表
- 回测时将因子打分表与回测底表交给 `backtest/` 中对应市场的回测函数
- 最终把有效因子沉淀到 `factors/`

这套流程的核心目标是：

- 保证研究与回测的数据口径稳定
- 保证研究项目之间的组织结构清晰
- 保证回测函数可复用
- 保证有效因子可以持续沉淀
